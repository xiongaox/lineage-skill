# CoursePackage

`CoursePackage` is the normalized middle layer between course distillation and Skill generation.

It lets the project support many kinds of courses without hardcoding a specific domain, teacher, or material structure.

## File

```text
<course>/course_package.json
```

Generated by:

```text
scripts/build_course_package.py
```

## Schema

```json
{
  "schema_version": "0.1",
  "manifest": {},
  "lessons": [],
  "concepts": [],
  "topics": [],
  "cases": [],
  "methods": [],
  "learning_checks": [],
  "quotes": [],
  "evidence": [],
  "study_paths": [],
  "boundaries": [],
  "quality": {}
}
```

## Fields

| Field | Purpose |
| --- | --- |
| `manifest` | Course name, source directory, generation time, source files. |
| `lessons` | Ordered lesson index with titles, summaries, duration, topics, source. |
| `concepts` | Terms, definitions, aliases, source lessons. |
| `topics` | Cross-lesson themes and topic relationships. |
| `cases` | Examples, demos, stories, applications. |
| `methods` | Frameworks, steps, procedures, decision rules. |
| `learning_checks` | Optional recall prompts, reflection checks, quizzes, or drills when suitable. |
| `quotes` | Memorable statements and source references. |
| `evidence` | Transcripts, visual analyses, screenshots, OCR documents, distillation files. |
| `study_paths` | Beginner, review, topical, time-boxed routes. |
| `boundaries` | Unsupported areas, confidence rules, professional boundaries. |
| `quality` | Completeness counts and missing recommended fields. |

## Design Rules

- Keep the schema course-agnostic.
- Preserve evidence paths.
- Do not force quizzes or exercises into courses where they do not fit.
- Treat OCR and summaries as evidence layers, not as unquestionable truth.
- Let Skill modes decide how to use the package.

## Relationship To Skill Modes

One `CoursePackage` can produce multiple Skills:

```text
CoursePackage
  ├─ course-expert
  ├─ study-coach
  ├─ practitioner
  ├─ citation-archive
  ├─ knowledge-base
  └─ domain-expert
```
